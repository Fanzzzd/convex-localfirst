Local-first tables are declared with the `lf` factory from
[`createLocalFirst`](/docs/getting-started). A table's declaration is its
**single source of truth**: the Convex schema derives from it (`.table()`), the
server sync config derives from it (`collectTables`), and the client runs the
same declaration locally (the provider's `modules`).

## `lf.table(name, options)`

```ts
import { v } from "convex/values";
import { lf } from "./localfirst";

export const todos = lf.table("todos", {
  shape: {                           // required: the row's fields (Convex validators)
    ownerId: v.string(),
    listId: v.string(),
    text: v.string(),
    done: v.boolean()
  },
  scope: lf.byUser("ownerId"),       // required
  idField: "localId",                // optional, default "localId"
  timestamps: true,                  // optional: adds createdAt/updatedAt, stamped automatically
  indexes: {                         // optional
    byList: ["ownerId", "listId", "createdAt"]
  }
});
```

| Option | Meaning |
| --- | --- |
| `shape` | **Required.** The table's fields. Don't declare the id field here — `.table()` adds it. |
| `scope` | **Required.** How rows are isolated and authorized — see [Scopes](#scopes). |
| `idField` | The stable, client-generated id column. Default `"localId"`. |
| `timestamps` | `true` adds `createdAt`/`updatedAt`; a tuple names them (e.g. `["created_at", "updated_at"]`). Stamped automatically on insert (both) and patch (updated). |
| `indexes` | Named indexes (field lists) your queries read from. Added to the Convex table by `.table()`. |
| `relations` | Declared client-side relations queried by name — see [Relations](#relations). |
| `setFields` / `counterFields` | Fields with convergent merges instead of LWW — see [Merges](#merges). |
| `searchFields` | Fields indexed for local full-text [search](/docs/search) (priority order; client-only). |
| `serverFields` | Names of fields the **server** mints — see [Server-minted fields](#server-minted-fields). |
| `clientCan` | A client-side mirror of `access.write` for [permission-aware UI](/docs/permissions). |

The returned object has the schema derivation `.table()` plus four function
builders: `query`, `insert`, `patch`, `remove`. Export the table **and** each
function — the functions are what your client references; the table object is
what `schema.ts` imports.

## `.table()` — the derived schema

```ts title="convex/schema.ts"
import { defineSchema } from "convex/server";
import { todos } from "./todos";

export default defineSchema({
  todos: todos.table()
});
```

`.table()` returns the Convex `TableDefinition`: your `shape` plus the id field
(`localId: v.string()`) plus every declared index. Row types flow from here —
`useQuery(api.todos.list, …)` infers the element type with no client generics.

Server-only fields that should exist in Convex but never sync locally go in
`extra`:

```ts
todos.table({ extra: { adminNote: v.optional(v.string()) } })
```

<Callout type="warn">
Local-first table functions are **not** directly callable on the server. They
exist in the deployment so the client can reference them by name, but invoking
the handler throws. Reads and writes flow through the client engine and sync via
`sync.push` / `sync.pull`.
</Callout>

## `query`

```ts
export const list = todos.query({
  args: { listId: v.string() },
  index: "byList",
  key: ({ auth, args }) => [auth.userId, args.listId],
  order: "asc",        // optional: "asc" | "desc"
  initial: []          // optional: value before the first local read resolves
});
```

`key` builds the index prefix to read — typically the scope value plus query
args. `auth.userId` is the resolved identity. In the browser the query runs
**locally** against the engine's derived view (rows match the index prefix your
`key` returns, sorted by the remaining index columns).

For a **byUser** table the same declaration also executes **on the server** when
invoked there — SSR loaders, scripts, `npx convex run`, plain Convex clients:
identity comes from `ctx.auth` and the declared index is walked with the same
`key` closure. It fails closed unless the index leads with the owner field and
the key pins it to the authenticated caller. byWorkspace/byProject queries
refuse server-side (membership lives in the sync config), and writes always
refuse — they must flow through the client so they sync.

Because the table knows its row type (from `shape`), `useQuery(api.todos.list, …)`
on the client infers the row type automatically — no client-side generics.
The `args` validators likewise type the `key` / `value` / `patch` closures.

## `insert`

```ts
export const create = todos.insert({
  args: { listId: v.string(), text: v.string() },
  value: ({ auth, args, now, localId }) => ({
    ownerId: auth.userId,
    listId: args.listId,
    text: args.text,
    done: false,
    createdAt: now,
    updatedAt: now
  })
});
```

`value` returns the new row. It receives `auth`, `args`, `now` (a timestamp), and
`localId` (the generated id for this row). Always set the scope field (here
`ownerId`) from `auth`/`args`, never from untrusted input — the server
re-enforces it anyway.

<Callout>
The closures run **on the client too** (that's what makes writes optimistic and
offline-capable), so they must stay pure: any expression over `auth` / `args` /
`now` / `localId` works — arbitrary computation included — but `ctx` throws on
access, and side effects belong in plain Convex functions.
</Callout>

## `patch`

```ts
export const toggle = todos.patch({
  args: { id: v.string(), done: v.boolean() },
  patch: ({ args, now }) => ({ done: args.done, updatedAt: now })
});
```

`patch` returns the changed fields only. The row id defaults to the `id` arg (or
one named after the table's `idField`); pass an explicit `id: ({ args }) => …`
closure only for a differently-named arg. Omit `patch` entirely to forward every
arg 1:1 (minus the id) — the common "update these fields" case.

## `remove`

```ts
export const remove = todos.remove({ args: { id: v.string() } });
```

## Scopes

A scope is **required** and defines both client-side partitioning and
server-side authorization.

### `lf.byUser(field)`

```ts
scope: lf.byUser("ownerId")
```

Rows belong to one user. The server forces `ownerId` to the authenticated user
on every push and only returns that user's rows on pull. The simplest, safest
default for personal data.

### `lf.byWorkspace({ workspaceIdField, membershipTable })`

```ts
scope: lf.byWorkspace({ workspaceIdField: "workspaceId", membershipTable: "ws_members" })
```

Rows belong to a workspace. Before accepting a write or returning a change, the
server checks the **same** `membershipTable` (read and write use one table) for a
row linking the user to the workspace. You provide that table and its membership
logic — see [Syncing](/docs/syncing#membership).

### `lf.byProject({ projectIdField, membershipTable })`

The same shape as `byWorkspace`, keyed by a project id. Use whichever name fits
your domain.

## Merges

Merges are **last-writer-wins per field** (not configurable — there is exactly
one policy): concurrent edits to different fields both survive; same-field
collisions resolve by arrival order at the server.

For convergent merges of specific fields, declare them per field:
`setFields` (array add/remove merge) and `counterFields`
(numeric increments accumulate). Both compose with the per-field merge. For real
collaborative editing of the *same* rich-text field (ordered lists, text), reach
for [document mode](/docs/document-mode).

## Relations

Declare relations **once**, next to the table, and query them by name — no
per-query combinators to repeat:

```ts
export const issues = lf.table("issues", {
  shape: { /* … */ state_id: v.string(), label_ids: v.array(v.string()), parent_id: v.optional(v.string()) },
  scope: byWorkspace,
  setFields: ["label_ids"],
  relations: {
    state:     lf.one("states", "state_id"),        // base[fk] -> target._id
    labels:    lf.many("labels", { via: "label_ids" }), // a setFields id-array -> targets
    subIssues: lf.backref("issues", "parent_id")     // reverse FK: target[fk] === base._id
  }
});
```

You then query them with `db.issues.scope({ workspace_id }).with("state",
"labels")`, fully typed. See [Typed client](/docs/typed-db#relations) for the
full story (and the per-query `.related()` / `.withRelations()` escape hatches).

## Server-minted fields

Some columns are minted by the **server** — an atomic `sequence_id` from
[`serverStamp`](/docs/syncing#row-level-access--server-minted-fields), or any
server-only value. List their names in `serverFields` so the client can strip
them from an [undo-of-delete](/docs/undo) re-insert (resurrecting a deleted row
re-mints them fresh, so re-sending the captured old value would be rejected):

```ts
lf.table("issues", { /* … */ serverFields: ["sequence_id"] });
```

`serverFields` is a client-only advisory (like `searchFields`); it pairs with
the matching `createSyncFunctions({ serverStamp })` field list on the server.

## Permission mirror

Declare `clientCan.write` to mirror the server's `access.write` on the client —
the same pure predicate, evaluated by [`useCan`](/docs/permissions) to
enable/disable UI. It is **advisory**; the server stays authoritative.

## Mixing with plain Convex

Any function **not** declared via `lf.table` is ordinary Convex. The same
`useQuery` / `useMutation` hooks call straight through to it (the
[fallback path](/docs/react-hooks#fallback)). Use plain Convex for server-only
actions, cross-tenant reads, and anything with heavy validation or side effects.
