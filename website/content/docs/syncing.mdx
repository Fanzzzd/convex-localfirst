The server side has two parts: a **component** you mount (no code), and your
**`sync.ts`** that calls `createSyncFunctions` (a dozen lines).

## The component

`convex-localfirst/component` is a real, mountable Convex component. It owns all
the sync bookkeeping so you never define those tables yourself:

- the **ledger** (idempotency — dedupes re-pushed ops),
- the append-only **change log** (the source of versions and pulls; deletes ride it
  too, as `kind: "delete"` changes — there is no separate tombstone table),
- the **id map** (local id ↔ server id),
- **row versions** (per-row version authority; survives change-log GC and drives
  snapshot bootstrap for cold clients).

Pull position (the cursor) is tracked **client-side** and sent on each pull — the
component keeps no per-client cursor state.

Mount it once:

```ts title="convex/convex.config.ts"
import { defineApp } from "convex/server";
import localfirst from "convex-localfirst/component/convex.config.js";

const app = defineApp();
app.use(localfirst);

export default app;
```

The parent app calls only the component's **public** functions via
`components.convexLocalFirst.*`. Convex codegen generates both your app's
`_generated` and the component's own `_generated`.

## `createSyncFunctions`

This builds the `push` mutation and `pull` query local-first clients sync
against. `collectTables` derives which tables are local-first — and each one's
scope / idField / timestamps — straight from your imported `lf.table` modules, so
the declaration lives in **one** place (the same modules the client's
[`modules`](/docs/react-hooks#the-provider) consumes). The helper wires app rows
to `ctx.db` and every bookkeeping operation to the mounted component.

```ts title="convex/sync.ts"
import { collectTables, createSyncFunctions } from "convex-localfirst/server";
import { components } from "./_generated/api";
import { mutation, query } from "./_generated/server";
import * as todos from "./todos";
import * as issues from "./issues";

export const { push, pull } = createSyncFunctions({
  component: components.convexLocalFirst,
  mutation,
  query,
  tables: collectTables({ todos, issues }),
  // Required only when a table is byWorkspace / byProject — see below.
  async isMember(ctx, { userId, scopeValue, membershipTable }) {
    const row = await ctx.db
      .query(membershipTable)
      .withIndex("by_user_ws", (q) => q.eq("userId", userId).eq("workspaceId", scopeValue))
      .unique();
    return row !== null;
  }
});
```

Adding a local-first table later is one more `import` + one key in
`collectTables`. (A hand-written `tables: { todos: { scope, idField } }`
record is also accepted, for the rare config that must differ from the `lf.table`
definition.)

| Option | Meaning |
| --- | --- |
| `component` | `components.convexLocalFirst` from your generated api. |
| `mutation` / `query` | The constructors from your `_generated/server`. |
| `tables` | Which app tables are local-first, and how each is scoped. |
| `isMember` | **Required if any table is `byWorkspace`/`byProject`.** The server-side membership check. |
| `schemaVersion` | Normally omitted — `collectTables` carries the version declared in `createLocalFirst`. |
| `changeRetentionMs` | How long change-log entries live before opportunistic GC (default 30 days; `Infinity` disables). |
| `visibility` | Per-table row-level read filter within an authorized scope — see below. |
| `serverStamp` | Per-table server-minted insert fields (sequence numbers) — see below. |
| `now` | Optional clock override (tests). |

## Row-level visibility & server-minted fields

Two per-table hooks cover the patterns scope membership alone can't:

**`visibility`** filters *rows* within a scope the user already has access to —
e.g. guests only see issues they created or are assigned:

```ts
visibility: {
  issues: async (ctx, { userId, row }) => {
    const m = await memberRow(ctx, userId, row.workspace_id);
    return m.role > GUEST || row.created_by === userId;
  }
}
```

It applies everywhere: bootstrap rows are filtered; an incremental change to a
row that *enters* visibility (say, the user gets assigned) arrives as a full-row
upsert (the client may lack the base row); one that *leaves* arrives as a delete
(the client evicts it); and writes obey "can't see → can't touch" — a patch or
delete of an invisible row is rejected.

**`serverStamp`** merges server-minted fields into every insert (client push
*and* `serverWriter`) — the pattern for atomic sequence numbers like `PROJ-123`:

```ts
serverStamp: {
  issues: async (ctx, { value }) => ({
    sequence_id: await nextCounter(ctx, `issue_seq:${value.project_id}`)
  })
}
```

It runs inside the push transaction, so the counter read-modify-write is
race-free under Convex OCC. Declare the stamped field in the table's `shape`
(as `v.optional(...)` so clients don't supply it) and it syncs back to every
client like any other field. A stamp may not touch the scope or id field.

## Cold start, GC, and revocation

The pull protocol handles the full client lifecycle without any extra wiring:

- **Snapshot bootstrap.** A client with no cursor for a scope gets the *current
  rows* (paged via `bootstrapCursors`), not a replay of history — so a
  year-old change log doesn't slow down a fresh device, and GC'd history is
  never needed. Pages are marked in `snapshotScopes`; the client evicts rows the
  snapshot didn't deliver only once the bootstrap completes, so an interrupted
  bootstrap never loses data.
- **Change-log GC.** Entries older than `changeRetentionMs` are pruned
  opportunistically during appends — no cron to set up. Row versions survive GC,
  so idempotency and bootstrap stay correct.
- **Doorbell.** The reactive `subscribe` path watches the pull query with
  `doorbell: true` — a content-free wake-up that reads at most one change per
  scope, so Convex reactivity stays cheap.
- **Revocation.** When membership is revoked, the next pull lists the scope in
  `deniedScopes`; the client evicts those rows and forgets the cursor.

## Schema changes: bump the version, clients reset

When a local-first table's shape changes incompatibly, bump the version — in
**one** place:

```ts title="convex/localfirst.ts"
export const lf = createLocalFirst({ schemaVersion: 2 });
```

That single number flows everywhere: the server sync gate (via `collectTables`)
rejects pushes and pulls from clients still on the old version
(`status.blockedBySchemaMismatch`, so stale clients pause instead of corrupting
data), and each upgraded client's default IndexedDB store is namespaced by the
new version — it starts clean and re-syncs from the server in full.

The trade-off is explicit: **unsynced offline writes made under the old version
are dropped** on upgrade (the server would reject them anyway). Additive changes
(a new `v.optional` field, a new table, a new function) don't need a bump — only
changes that make old rows or old ops unreadable do.

## Membership (I7)

For `byWorkspace` / `byProject` tables, **the server** decides access — never the
client. Your `isMember` callback runs before any workspace write is accepted or
any change is returned. If a `byWorkspace`/`byProject` table is declared without
an `isMember`, `createSyncFunctions` throws a clear error rather than silently
allowing access.

## Trust model & current limitations

Be explicit about what the server does and does not guarantee — this is a
client-authoritative local-first design, like Replicache, not a server that
re-runs your mutations.

**The server enforces (the security boundary):**

- **Identity** from `ctx.auth` (never the client), per [I7](/docs/reference/invariants).
- **Scope / membership**: a write is accepted only for a scope the caller owns
  (`byUser`) or is a member of (`byWorkspace`/`byProject`); for a patch/delete the
  scope is read from the **stored row**, never from client-supplied values.
- **Partition integrity**: a patch may not move a row across scopes (change its
  `workspaceId`/`ownerId`/`projectId`) or rewrite its id field.
- **Schema/types**: writes go through Convex's own schema validators, so a field
  can't be the wrong type or undeclared (if your schema is strict).

**The server does *not* (yet) enforce — design these around it:**

- **Business rules.** A local-first mutation's `plan()` runs on the *client*; the
  server applies the resulting insert value / patch after the checks above — it
  does **not** re-run your mutation body. So within an authorized scope a client
  can write any *schema-valid* field value. Encode hard invariants as Convex schema
  validators; server-side mutation re-execution is a future milestone. Treat the
  schema as your server-side validation surface.
- **Concurrent-write recency.** Same-field writes are last-writer-wins by
  **arrival order** at the server, so a stale offline write can overwrite a newer
  edit of the *same field*. Convergent per-field merges (`setFields`,
  `counterFields`) never clobber. Client-side rebase
  ([I1](/docs/reference/invariants)) keeps each device's *local* view consistent
  throughout.

## What runs underneath

`createSyncFunctions` is a thin adapter over the pure `serverSync` engine —
`handlePush` / `handlePull`, driven by a `SyncConfig` and a `ServerStore`
interface. The engine is deliberately runtime-agnostic (no direct `ctx.db`
dependency) so it stays testable and portable; the adapter binds it to `ctx.db`
plus the component.

If you need full control (custom storage, batching component writes, extra
validation), you can implement `ServerStore` yourself and call `handlePush` /
`handlePull` directly — `createSyncFunctions` is the turnkey path, not the only
one. Both are exported from `convex-localfirst/server`.

<Callout title="Auth in production">
The helper derives identity from `ctx.auth` and falls back to a supplied id only
for local demo backends with no auth provider. In production, auth is what
enforces [invariant I7](/docs/reference/invariants).
</Callout>
